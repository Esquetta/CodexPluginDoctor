# v0.17 Attestation Plan

## Goal

Add a local attestation artifact that lets plugin authors prove what Codex Plugin Doctor validated before a package is published or shared.

The feature should strengthen trust without introducing key management, hosted signing, or remote verification before the CLI contract is mature enough for 1.0.

## Command

```bash
codex-plugin-doctor doctor attest <path> [--json] [--output <path>]
```

## Artifact Contract

The JSON artifact should include:

- `schemaVersion`
- `kind`
- `generatedAt`
- `version`
- `targetPath`
- `subject`
- `packageFingerprint`
- `reportDigest`
- `summary`
- `verification`

The `packageFingerprint` proves the local package contents that were evaluated. The `reportDigest` proves the validation, security, compatibility, recommendations, and trust signals used to make the attestation. The artifact is intentionally unsigned in v0.17, but it is structured so a future release can add signature fields without changing the command shape.

## Data Sources

Reuse the shared package analysis pipeline so attestation reflects the same validation, security, compatibility, recommendations, and trust signals used by:

- `doctor recommend`
- `doctor trust`
- `doctor export --bundle`
- `doctor perf`

## Safety

- Do not execute package lifecycle scripts.
- Do not run runtime probes automatically.
- Exclude common transient dependency/build directories from package fingerprinting.
- Keep the text output human-readable and the JSON output stable enough to become part of the 1.0 contract.

## Verification

- Add focused unit tests for JSON artifact shape, deterministic fingerprinting, output-file writing, and missing-output-path handling.
- Run `npm test`.
- Run `npm run build`.
- Run a CLI smoke command against `examples/codex-doctor-runtime`.
