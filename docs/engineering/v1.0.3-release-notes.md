# v1.0.3 Release Notes

`v1.0.3` adds a consolidated release evidence command for maintainers who want one signed artifact before publishing a Codex plugin or MCP package.

## Highlights

- `doctor release-evidence <path> --sign-key-env NAME` builds one JSON bundle with attestation, attestation verification, validation corpus, performance, security, trust, package metadata, and git release gates.
- Release evidence uses env-only HMAC keys; inline `--sign-key` is rejected for this command.
- Strict release gates require a resolved git commit, exact git tag, and clean worktree before `releaseReady` can be true.
- `--allow-dirty` and `--allow-untagged` are available for local rehearsal when an operator intentionally wants non-release evidence.
- `doctor.release.evidence` is now listed in `doctor contract --json` as a public machine-readable surface.
- Shared release evidence JSON uses the same redaction path as export bundles for token-like values.

## Verification

Run these checks before publishing:

```bash
npm test
npm run build
$env:DOCTOR_SIGNING_KEY = "local-release-secret"
node dist/cli.js doctor release-evidence examples/codex-doctor-runtime --json --sign-key-env DOCTOR_SIGNING_KEY --allow-dirty --allow-untagged --max-total-ms 5000
```

For final release evidence after tagging, omit `--allow-dirty` and `--allow-untagged` so git release gates must pass.
