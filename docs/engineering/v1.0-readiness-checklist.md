# v1.0 Readiness Checklist

## Purpose

This checklist is the stable release gate for moving Codex Plugin Doctor from the `0.x` line to `1.0.0`.

The goal is to prove that the CLI, public package, GitHub Action, and machine-readable contracts are stable enough for users to depend on without changing the product scope at the last minute.

## Stable Release Target

- Next target: `1.17.0` minor release on the stable 1.x line
- Release lane: stable package publication, docs accuracy, and contract verification
- Feature policy: No new feature work unless a blocker is discovered

## Scope Boundary

In scope for the active `1.0.0` stable release:

- README and docs accuracy
- release metadata correctness
- npm and GitHub Release sync
- install and smoke verification
- output contract verification
- GitHub Action artifact verification
- known limitations documentation

Out of scope for the active `1.0.0` stable release:

- hosted dashboard
- marketplace behavior
- hosted history or team sharing
- broad governance positioning
- new client families beyond the current compatibility matrix
- default runtime probing

## Known Limitations To Keep Explicit

- Codex Plugin Doctor remains Codex-first; broader MCP Doctor expansion is a post-1.0 path.
- Runtime probing is opt-in because it starts package-local MCP servers.
- Attestations and release evidence can be locally signed with an HMAC key supplied through `--sign-key-env`; key management remains local to the operator.
- GitHub Action artifact generation is CI-friendly, but consumers still own retention and Code Scanning upload policy.
- The CLI validates package quality; it does not certify marketplace approval.

## Required Smoke Commands

Run from a clean checkout before the active `1.0.0` stable release:

```bash
npm ci
npm run release-check
node dist/cli.js self-test
node dist/cli.js doctor corpus
node dist/cli.js doctor contract --json --output output-contract.json
node dist/cli.js check examples/codex-doctor-runtime --runtime --no-animations
```

The npm preinstall scanner should also be smoke tested against the CLI package itself, but that command is expected to return a non-zero status because `codex-plugin-doctor` is a validator package, not a Codex plugin package:

```bash
node dist/cli.js doctor npm codex-plugin-doctor --json --output npm-preinstall.json
```

Expected result: `schemaVersion: "1.0.0"`, `kind: "doctor.npm"`, package version resolved from npm, and `plugin.manifest.missing`.

Run after registry publication of the stable release:

```bash
npm install -g codex-plugin-doctor@latest
codex-plugin-doctor --version
codex-plugin-doctor self-test
codex-plugin-doctor doctor corpus
npm run verify-release-sync
```

For the published stable package, run the same expected non-plugin package scan separately:

```bash
codex-plugin-doctor doctor npm codex-plugin-doctor@latest --json --output npm-preinstall.json
```

Expected result: non-zero status with `plugin.manifest.missing`, not a crash or malformed JSON report.

## GitHub Action Smoke

Validate that the documented workflow still maps to `action.yml`:

- `policy: codex-publish`
- `json: "true"`
- `markdown: "true"`
- `sarif: "true"`
- `upload-artifact: "true"`
- `step-summary: "true"`
- report outputs: `status`, `report-dir`, `summary-path`, `json-path`, `sarif-path`

## Contract Checks

Before the active `1.0.0` stable release, confirm:

- `doctor contract --json` reports `stable-through-1.0`
- existing rule IDs are not renamed
- existing default severities are not changed without an explicit breaking decision
- public JSON schema surfaces retain `schemaVersion: "1.0.0"`
- new surfaces are additive and documented

## Stable Release Exit Criteria

The active `1.0.0` stable release is ready when:

- `npm run release-check` passes
- GitHub Actions passes on `main`
- `npm run verify-release-sync` passes after publication
- npm `latest` points to the stable release
- registry install smoke passes
- no stale `0.1.x` release-state language remains in active docs
- known limitations are visible from the README/docs path
- no new feature work was added during stable release prep

## Final 1.0 Promotion Criteria

`1.0.0` is ready when:

- the final release candidate has no blocker feedback
- no public contract corrections are required
- npm `latest` install smoke remains healthy
- GitHub Action artifact smoke remains healthy
- release notes clearly state compatibility and known limitations
